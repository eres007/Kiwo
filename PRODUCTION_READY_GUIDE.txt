PART 3: HOW USERS USE KIWO (Complete User Flow)
═══════════════════════════════════════════════════════════════════════════════

STEP 1: DISCOVER & SIGNUP
─────────────────────────
1. User visits website (deployed on Vercel/Render)
2. User sees Hero section with value proposition
3. User enters email in signup form
4. Form submits to: POST /api/auth/signup
5. Backend creates user account in Supabase
6. User receives confirmation (check email message)

STEP 2: LOGIN & AUTHENTICATE
─────────────────────────────
1. User receives email with login link or instructions
2. User logs in via: POST /api/auth/login
3. Backend returns JWT token
4. User stores JWT in browser/app
5. JWT used for all subsequent API calls

STEP 3: CAPTURE MEMORIES
────────────────────────
1. User interacts with AI tools (Cursor, Claude, ChatGPT, etc.)
2. User tells AI about their project, preferences, decisions
3. MCP Server or File Sync captures this context
4. Raw context sent to: POST /api/memory/capture
5. Backend receives raw memory data

STEP 4: AI REFINEMENT
─────────────────────
1. Backend sends raw memory to Qwen3 AI model
2. AI extracts structured information:
   - Project details
   - Tech stack decisions
   - Coding preferences
   - Current tasks
   - Architecture decisions
3. Refined memory stored in database with vector embedding

STEP 5: MEMORY STORAGE
──────────────────────
1. Refined memory stored in Supabase PostgreSQL
2. Vector embedding created for semantic search
3. Metadata stored: importance score, category, timestamps
4. Encrypted at rest (AES-256)
5. Indexed for fast retrieval

STEP 6: MEMORY INJECTION
────────────────────────
1. User opens new AI tool session
2. Retrieval engine queries: GET /api/memory/retrieve
3. Semantic search finds relevant memories
4. Top memories filtered by importance score
5. Memories injected into new session automatically
6. User's AI tool now has full context

STEP 7: ONGOING SYNC
────────────────────
1. File sync watches project directory
2. Auto-updates .cursorrules, CLAUDE.md, etc.
3. Chrome extension injects memories in browser
4. MCP protocol keeps all tools synchronized
5. User never re-explains context again

PART 5: API ENDPOINTS USERS CAN ACCESS
═══════════════════════════════════════════════════════════════════════════════

PUBLIC ENDPOINTS (No JWT Required):
POST   /api/auth/signup              - Create new account
POST   /api/auth/login               - Login with email/password
POST   /api/auth/refresh             - Refresh JWT token
GET    /health                       - Health check

PROTECTED ENDPOINTS (JWT Required):
POST   /api/memory/capture           - Capture new memory
GET    /api/memory/retrieve          - Get relevant memories
GET    /api/memory/list              - List all memories
PUT    /api/memory/:id               - Update memory
DELETE /api/memory/:id               - Delete memory
GET    /api/memory/search            - Search memories
POST   /api/memory/export            - Export all memories (GDPR)
DELETE /api/memory/delete-all        - Delete all memories (GDPR)
GET    /api/monitoring/metrics       - View system metrics
GET    /api/monitoring/backup-health - Check backup status
